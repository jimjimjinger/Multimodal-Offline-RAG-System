# 상황 인지형 G4 개선 설계

## 목적

현재 정식 실험 결과로 사용할 수 있는 것은 `07_pilot_results.md`에 정리된 G3 멀티모달 RAG 결과이다.

G4는 아직 정식 구현 단계가 아니며, 현재는 실습 단계 라벨을 단순히 검색어와 re-ranking 점수에 반영해 본 예비 실험만 수행한 상태이다. 이 예비 실험에서는 G3 대비 성능 개선이 나타나지 않았다.

따라서 본 문서는 기존 결과를 보고하는 문서가 아니라, 앞으로 정식 G4 상황 인지형 멀티모달 RAG를 어떻게 설계하고 개선할지 정리하는 계획 문서이다.

## 현재 구조

현재 구현된 G3 멀티모달 RAG는 다음 흐름으로 동작한다.

1. 사용자의 질문을 BGE-M3로 임베딩한다.
2. ChromaDB의 텍스트 컬렉션에서 관련 매뉴얼 chunk를 검색한다.
3. 텍스트 검색 결과에 연결된 이미지 후보를 수집한다.
4. 검색된 텍스트 page 주변의 이미지 후보를 추가한다.
5. 이미지 전용 ChromaDB 컬렉션에서 관련 이미지 후보를 검색한다.
6. 이미지 검색 점수, 텍스트 순위 점수, page 근접도, SigLIP 기반 매핑 점수를 통합해 이미지 Top-10을 정렬한다.

G3의 70개 질의셋 기준 1차 결과는 다음과 같다.

| 지표 | G3 결과 |
|---|---:|
| Text Recall@1 | 81.4% |
| Text Recall@5 | 95.7% |
| Text Recall@10 | 100.0% |
| Text MRR | 0.879 |
| Image Recall@1 | 32.9% |
| Image Recall@5 | 70.0% |
| Image Recall@10 | 75.7% |
| Image MRR | 0.485 |

이 결과는 `SCIE용/07_pilot_results.md`에 정리되어 있다.

## 예비 실험에서 확인한 한계

실습 단계 라벨을 단순히 `실습 단계 + 질문` 형태로 검색어에 추가하고, 후보 이미지의 heading/page/주변 텍스트와 라벨 단어가 겹치는 정도를 점수에 반영해 보았다.

하지만 이 방식은 정식 G4라고 보기 어렵다.

이유는 다음과 같다.

1. 실습 단계 라벨이 실제 매뉴얼의 page 범위와 직접 연결되어 있지 않다.
2. section heading과 실습 단계 사이의 대응 관계가 명시적으로 정의되어 있지 않다.
3. 단계별 핵심 keyword 사전이 없어서 관련 없는 후보에도 점수가 들어갈 수 있다.
4. 질문별 정답 이미지는 사용하지 않았지만, 상황 정보가 검색 범위를 충분히 좁히지 못했다.

따라서 현재의 단순 라벨 방식은 상황 인지형 검색이라기보다, 라벨 기반 보조 re-ranking 파일럿에 가깝다.

## 개선 방향

정식 G4는 단순히 실습 단계 라벨을 검색어에 붙이는 방식이 아니라, 실습 단계가 검색 범위와 후보 순위에 구조적으로 반영되는 방식이어야 한다.

이를 위해 다음 요소를 추가한다.

| 추가 요소 | 목적 |
|---|---|
| 실습 단계별 page 범위 | 현재 단계와 관련된 매뉴얼 범위를 제한하거나 가중 |
| 실습 단계별 section heading | 관련 section의 텍스트와 이미지 후보를 우선 |
| 실습 단계별 keyword 사전 | 질문과 후보 이미지의 단계 적합성 계산 |
| 상황 적합성 점수 | G3 점수에 추가되는 G4 전용 점수 |
| G3 vs G4 비교표 | 상황 정보가 실제 성능을 개선하는지 검증 |

## 추가할 매핑표

가장 먼저 구축해야 하는 것은 실습 단계별 매핑표이다.

예상 파일:

- `SCIE용/excel/09_stage_context_map.xlsx`
- `SCIE용/data/09_stage_context_map.csv`
- `data/processed/stage_context_map.json`

매핑표 구조는 다음과 같이 구성한다.

| 실습 단계 | 관련 page 범위 | 관련 section | 핵심 keyword | 비고 |
|---|---|---|---|---|
| 툴 플랜지/I/O 배선 | 127-132 | Schematic Diagram | 툴 플랜지, X1, 핀맵, Digital Output | 플랜지 커넥터/핀맵 |
| 안전 I/O/비상정지 배선 | 145-148 | Safety I/O | EM, TBSFT, Emergency Stop, 비상정지 | 안전 입력 단자 |
| 프로그래밍/모션 명령 | 272-276 | Motion | MoveJ, MoveL, Motion, 직선 이동 | 모션 명령 |
| 안전 설정/공간 제한 구역 | 226-238 | Safety Settings | Zone, Space Limit, 안전 구역 | 안전 구역 설정 |

주의할 점은 질문 번호나 정답 이미지 파일명을 매핑표에 넣지 않는 것이다.

좋은 매핑:

```text
툴 플랜지/I/O 배선 -> page 127-132, Schematic Diagram, X1, 핀맵
```

사용하면 안 되는 매핑:

```text
Q05 -> page_128_img_0_0.jpeg
```

후자는 정답을 직접 알려주는 방식이므로 실험 누수에 해당한다.

## 수정할 검색 구조

정식 G4에서는 기존 G3 점수에 상황 적합성 점수를 추가한다.

현재 G3 점수는 다음 요소를 사용한다.

```text
G3 이미지 점수 =
이미지 검색 점수
+ 텍스트 검색 순위 점수
+ page 근접도 점수
+ 텍스트-이미지 매핑 점수
+ SigLIP 기반 이미지 점수
```

G4에서는 여기에 다음 점수를 추가한다.

```text
G4 이미지 점수 =
G3 이미지 점수
+ 실습 단계 page 범위 일치 점수
+ 실습 단계 section 일치 점수
+ 실습 단계 keyword 일치 점수
```

예를 들어 현재 실습 단계가 `툴 플랜지/I/O 배선`이면 page 127-132 범위의 이미지, `Schematic Diagram` section, `X1`, `핀맵`, `Tool Flange` keyword가 포함된 후보에 가중치를 준다.

반대로 현재 단계와 관련 없는 page/section의 이미지는 점수를 올리지 않거나 낮은 우선순위로 유지한다.

## 구현 계획

1. 70개 질의셋의 `실습 단계` 목록을 확정한다.
2. 각 실습 단계별 관련 page 범위를 매뉴얼 기준으로 지정한다.
3. 각 실습 단계별 section heading과 keyword를 작성한다.
4. 매핑표를 CSV/XLSX/JSON으로 저장한다.
5. `rag_search.py`에서 stage context map을 로드한다.
6. 이미지 후보별 page/section/keyword 일치 점수를 계산한다.
7. 기존 G3 점수와 상황 적합성 점수를 결합한다.
8. 70개 질의셋으로 G3와 G4를 다시 비교한다.

## 평가 계획

정식 G4 구현 후에는 G3와 G4를 같은 70개 질의셋으로 비교한다.

| 비교 | 목적 |
|---|---|
| G3 멀티모달 RAG | 현재 기준 성능 |
| G4 상황 인지형 멀티모달 RAG | 실습 단계 정보 반영 후 성능 |

평가 지표는 다음을 사용한다.

| 지표 | 의미 |
|---|---|
| Image Recall@1 | 정답 이미지가 1순위인지 |
| Image Recall@5 | 정답 이미지가 Top-5 안에 포함되는지 |
| Image Recall@10 | 정답 이미지가 Top-10 안에 포함되는지 |
| Image MRR | 정답 이미지가 얼마나 앞쪽에 배치되는지 |

목표는 G3 대비 G4에서 Recall@5 또는 MRR이 개선되는지 확인하는 것이다.

## 기대 효과

정식 G4가 잘 동작하면 다음 효과를 기대할 수 있다.

1. 현재 실습 단계와 관련 없는 이미지 후보를 줄일 수 있다.
2. 같은 질문이라도 실습 단계에 맞는 이미지가 더 상위로 올라올 수 있다.
3. 단순 멀티모달 RAG가 아니라 상황 정보를 활용한 검색 개선으로 논문 기여를 확장할 수 있다.

다만 성능 향상은 매핑표 품질에 따라 달라질 수 있으므로, G4는 구현 후 반드시 G3와 정량 비교해야 한다.
